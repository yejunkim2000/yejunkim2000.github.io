> 보안 컨퍼런스 NDSS 2024의 ChatAFL을 중심에 두고, 선행 연구 2편과 후속 연구 2편을 함께 정리한다. 프로토콜 퍼징의 지식 원천이 관측에서 LLM 사전학습으로, 다시 외부 문서 기반 근거로 이동하는 흐름을 다룬다.

## 다루는 논문

| 구분 | 논문 | 발표 | 기여 |
|------|------|------|------|
| 선행 ① | AFLNet | ICST 2020 | 명세 없이 상태를 학습하는 그레이박스 프로토콜 퍼저 |
| 선행 ② | TitanFuzz | ISSTA 2023 | LLM을 입력 생성기로 사용 |
| 중심 | ChatAFL | NDSS 2024 (Distinguished Paper) | 두 접근의 결합 |
| 후속 ① | MultiFuzz | arXiv 2025 (preprint) | 환각 문제를 근거 기반으로 보완 |
| 후속 ② | ChatFuMe | arXiv 2025 (preprint) | 토큰 비용 문제를 모델 합성으로 보완 |

중심 논문은 보안 4대 학회(NDSS)의 Distinguished Paper다. 선행 두 편은 테스팅·소프트웨어공학 학회(ICST·ISSTA) 발표작이지만 각 접근의 시초다. 후속 두 편은 2025년 arXiv 프리프린트로 피어리뷰 전이며, 인용한 수치는 저자 주장이다.

## 1. 문제 정의 — 프로토콜 퍼징의 두 가지 벽

네트워크 프로토콜(RTSP, MQTT, FTP 등)을 사용하는 서버·IoT·미들웨어는 공격 표면이 넓지만, 퍼징을 어렵게 하는 두 가지 구조적 제약이 있다.

**상태의 벽.** 프로토콜은 상태기계다. `LOGIN → SETUP → PLAY` 같은 순서를 지켜야 깊은 코드 경로에 도달한다. 단일 패킷을 무작위로 던지는 퍼저는 초기 상태에서 더 나아가지 못한다.

**구조의 벽.** 메시지는 엄격한 문법을 갖는다. 무작위 변이는 파서 진입 단계에서 거부되어 실행 예산을 소모한다.

여기에 더해, 기계가 읽을 수 있는 명세(spec)가 있으면 퍼징이 수월하지만 현실의 구현체에는 명세가 없다는 제약이 있다. 다섯 편의 논문은 모두 "명세 없는 환경에서 퍼징에 필요한 지식을 어디서 얻을 것인가"라는 질문에 대한 서로 다른 답이다.

## 2. 선행 연구

### 2.1 AFLNet (ICST 2020)

*Pham, Böhme, Roychoudhury*

상태기반 프로토콜을 위한 최초의 그레이박스(커버리지 피드백) 퍼저다. 세 가지 설계가 핵심이다.

- pcap 캡처를 시드로 사용해 명세 의존성을 제거
- 입력을 메시지 단위로 구조화하고 변이
- 서버 응답 코드를 상태로 간주하여 상태기계를 역추론

가장 중요한 기여는 알고리즘이 아니라 문제의 재정의다. 명세 대신 관측된 동작에서 상태를 학습한다는 접근을 제시했다.

**한계.** 시드에 없던 새로운 구조의 메시지를 합성하지 못하고(구조의 벽), 상태 해상도가 거칠며, 커버리지 정체 시 탈출 전략이 없다.

### 2.2 TitanFuzz (ISSTA 2023)

*Deng, Xia, Peng, Yang, Zhang*

LLM을 퍼징 입력 생성기로 직접 사용한 최초의 연구다. 생성형 모델(Codex)과 인필링 모델(InCoder)로 딥러닝 라이브러리의 입력을 생성·변이했다. TensorFlow에서 30.38%, PyTorch에서 50.84%의 커버리지 향상과 버그 65개(신규 44개)를 보고했다.

핵심 통찰은 LLM이 입력 구조에 대한 지식을 이미 사전학습으로 보유하고 있다는 점이다.

**한계.** 무상태(stateless) 방식이다. 독립적인 프로그램 하나를 생성할 뿐, 상태 시퀀스 개념이 없다(상태의 벽).

두 선행 연구는 상호 보완적이다. AFLNet은 상태를 다루지만 구조 합성이 약하고, TitanFuzz는 구조를 다루지만 상태 개념이 없다. 두 접근을 결합할 여지가 이 시점에 이미 존재했다.

## 3. 중심 연구 — ChatAFL (NDSS 2024)

*Meng, Mirchev, Böhme, Roychoudhury (Distinguished Paper)*

**목표.** 명세 없는 구현체에서 상태의 벽과 구조의 벽을 동시에 넘는다. AFLNet의 퍼징 골격에 TitanFuzz의 LLM 지식을 결합하는 전략을 취한다.

**핵심 설계 — 역할 분담.** 퍼저는 실행·피드백 엔진을, LLM은 지식·계획 엔진을 담당한다. 퍼징 루프를 새로 설계하지 않고, 진행이 막히는 세 지점에서만 LLM을 호출한다.

| 개입 지점 | 역할 | 대응하는 벽 |
|---|---|---|
| 문법 구성 | 메시지 타입별 문법 추출 후 구조 인지 변이 | 구조의 벽 |
| 시드 다양화 | 기록된 시퀀스의 메시지 다양성 확장 | 초기 시드 부족 |
| 시퀀스 예측 | 정체 시 다음 메시지·상태 전이 예측 | 상태의 벽 |

**평가 결과 (ProFuzzBench 기준)**

| 지표 | vs AFLNet | vs NSFuzz |
|------|-----------|-----------|
| 상태 전이 | +47.6% | +42.7% |
| 발견 상태 | +29.6% | +25.8% |
| 코드 커버리지 | +5.8% | +6.7% |
| 신규 취약점 | 9개 | 각 3, 4개 |

코드 커버리지 향상은 5.8%로 크지 않다. 그러나 상태 전이 47.6% 증가와 신규 취약점 9개 발견이 보여주듯, 이 연구의 기여는 단순히 더 많은 코드를 실행한 것이 아니라 기존에 도달할 수 없던 상태 공간을 열었다는 데 있다. 어떤 지표로 성과를 측정하느냐가 기여의 성격을 결정하는 사례다.

**한계.** ① LLM 환각, ② 대상 프로토콜에 대한 LLM의 지식을 전제한다는 점, ③ 호출마다 발생하는 토큰 비용, ④ 단발성 프롬프트의 신뢰성. 이 네 가지가 후속 연구의 출발점이 된다.

## 4. 후속 연구

### 4.1 MultiFuzz (2025) — 근거 기반 보완

단일 LLM이 프로토콜을 정확히 안다는 가정에서 환각이 발생한다는 문제를 다룬다. 세 가지로 대응한다.

- **멀티에이전트:** 문법 추출, 시드 강화, 정체 탈출을 각각 분업
- **RAG:** RFC 문서를 명제 단위로 분할해 벡터DB에 저장하고 실제 명세를 인출
- **CVE 주입:** NVD의 실제 취약점 데이터를 활용

RTSP/Live555 대상 24시간 실험에서 ChatAFL 대비 분기 커버리지 0.9%, 상태 전이 2.3% 향상에 그쳤다(AFLNet 대비로는 94.5%). ChatAFL 대비 증분이 작다는 사실은 커버리지가 천장에 근접했고, 경쟁의 초점이 양적 향상에서 근거성이라는 질적 측면으로 이동했음을 시사한다.

### 4.2 ChatFuMe (2025) — 비용 상각

*Huang, Wang, Zhou*

ChatAFL이 테스트케이스마다 LLM을 호출해 토큰 비용이 급증하는 문제를 다룬다. 접근을 전환하여, LLM에게 개별 입력이 아니라 메시지 시퀀스를 생성하는 프로그램(모델)을 만들게 한다. 한 번의 호출로 재사용 가능한 생성기를 확보하고, 피드백으로 모델을 수정한다.

결과적으로 토큰 사용량을 216,596 토큰/160 호출에서 5,980 토큰/16 호출로, 약 36배 절감했다. MQTT·Modbus·DAAP 6개 구현에서 버그 12개를 발견했다. LLM 호출 단위를 테스트케이스 단위에서 프로그램 단위로 바꾸면서, 비용이 1급 평가 지표로 부상했다.

## 5. 연구 흐름 정리

```
   [상태의 벽]                          [구조·지식의 벽]
   AFLNet (2020)                        TitanFuzz (2023)
   지식 원천 = 실행 관측                  지식 원천 = LLM 사전학습
   상태 O / 구조 X                       구조 O / 상태 X
        \____________ 수렴 ____________/
                      |
              ChatAFL (NDSS 2024)
       AFLNet 루프 + LLM 3개입 / 상태 전이 +47.6%
                      |
        +-------------+--------------+
   MultiFuzz                      ChatFuMe
   멀티에이전트 + RAG + CVE        모델 합성, 토큰 36배 감소
   양 -> 질 (근거성)               정확도 -> 비용
```

두 독립적 접근이 ChatAFL에서 수렴한 뒤, 각 약점에 따라 다시 분화하는 구조다. 통합 논문 이후 정제 단계로 접어드는 전형적 패턴이다.

## 6. 관찰되는 동향

1. **지식 원천의 이동.** 관측(AFLNet) → LLM 내부 지식(ChatAFL) → 외부 문서·CVE(MultiFuzz)로 지식의 출처가 세 번 바뀌었다.
2. **LLM 호출 단위의 확대.** 테스트케이스 단위에서 프로그램 단위로(ChatFuMe). 토큰 수가 평가 표에 등장하기 시작했다.
3. **환각 대응 = 그라운딩.** 단일 프롬프트에서 멀티에이전트와 RAG로 이동. 생성 결과를 신뢰하지 않고 인출·검증하는 방향이다.
4. **평가지표 포화.** 후속 연구의 커버리지 증분이 ChatAFL 대비 약 1%에 그치며 ProFuzzBench가 천장에 도달했다. 퍼징 평가 방법론 논의(Klees 2018, SoK 2024)와 직결되며, 버그·CVE 발견과 비용 효율이 새 지표로 채택되고 있다.
5. **퍼저 정체성의 재정의.** 변이기였던 퍼저가 실행·피드백 엔진으로, LLM이 지식·계획 엔진으로 분리됐다. ChatAFL이 그 원형이다.

## 7. 전망

- **비용 상각의 주류화.** 모델·프로그램 합성과 소형 로컬 모델 증류. 실무 병목이 정확도보다 토큰·지연 비용에 있기 때문이다.
- **그라운딩의 폐루프화.** 현재 RAG는 입력 측(RFC를 프롬프트로)에 적용된다. 다음 단계는 출력 측 검증이다. LLM이 생성한 메시지를 실제 서버 응답·파서 피드백으로 즉시 반증해 환각을 자동 폐기하는 구조다.
- **에이전트형 자율 퍼징.** 가설·테스트·학습 루프를 자율적으로 수행하는 에이전트. Big Sleep, DARPA AIxCC 흐름과 합류할 가능성이 있다.
- **멀티모달 명세 융합.** RFC 텍스트, pcap 트래픽, 소스 코드, CVE를 함께 입력으로 받는 모델.
- **벤치마크 재정의.** ProFuzzBench 포화와 데이터 누수(사전학습이 대상 CVE를 이미 학습한 문제)로 인해, 견고하고 비용을 정규화한 벤치마크가 필요하다.
- **이중 용도 문제.** 동일 기법이 방어 측에서는 자동 결함 발견, 공격 측에서는 자동 익스플로잇 정찰로 쓰인다. 특히 CVE 그라운딩은 오용 시 무기화가 쉬워 책임 공개와 레이트리밋이 규범으로 부상하고 있다.

ChatAFL은 LLM 기반 프로토콜 퍼징의 실현 가능성을 입증했다. 이후 연구의 초점은 비용 효율성, 근거성, 자율성으로 옮겨가고 있다.
