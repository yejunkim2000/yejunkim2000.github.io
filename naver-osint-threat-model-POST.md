> **윤리 고지:** 이 글은 공격 매뉴얼이 아니라 **방어를 위한 외부 공격 표면 지도**다. 대상은 **공식 버그바운티를 운영하는 실존 기업 네이버(NAVER)**이며, 네이버는 외부 보안 연구를 공식적으로 허용·장려한다. 본 글은 공개 정보·공개 표준·일반적인 보안 지식만 사용했고, 활성 스캔·익스플로잇·침투는 일절 다루지 않는다. 모든 취약점은 "구조상 가능한 가설"일 뿐 실제 존재가 확인된 결함이 아니며, 실제 검증은 정식 버그바운티 규정(자동스캔·DoS·사용자 데이터/소스 열람 금지) 범위 안에서만 가능하다.
---
## 들어가며
위협 모델링(Threat Modeling)은 "공격자의 눈으로 우리 시스템을 보는 것"이다. 실제 공격 없이, 공개된 정보와 구조적 추론만으로 어디가 약한지 지도를 그리는 작업이다.
이 글에서는 멀티서비스 플랫폼 **네이버**(검색·메일·결제·클라우드·커머스 등을 운영)를 대상으로, OSINT 기반 위협 모델링을 처음부터 끝까지 수행한다. 특정 회사를 깎아내리려는 것이 아니라 **방법론과 프레임워크 적용 과정**을 보여주는 것이 목적이며, 네이버가 공개 버그바운티를 운영한다는 점에서 이런 외부 관점 분석은 회사가 장려하는 활동의 문서화 버전에 해당한다.
---
## 핵심 메시지 3줄
1. **폭발 반경 중심 사고** — 멀티서비스 회사의 위험은 결함 *개수*가 아니라 신뢰축 결함의 *전파 반경*이다. 1순위 방어는 인증축과 교차서비스 상관분석이다.
2. **Red ↔ Blue 대칭** — 모든 공격 단계에는 대응하는 탐지 신호가 있다. 위협 모델은 곧 탐지 백로그다.
3. **합법성의 우위** — 공개정보·가설·버그바운티 규정 안에서만 정당하다.
---
## 1. 대상 선정과 기본 조사
위협 모델링의 첫 단계는 **공격 표면의 경계를 정하는 것**이다. 버그바운티를 운영하는 회사라면, BB 스코프 자체가 "회사가 공식적으로 인정한 공개 자산 지도" 역할을 한다. 네이버 역시 공개 버그바운티 스코프를 제공하므로 이를 출발점으로 삼는다.
네이버의 공개 자산 구성(버그바운티 스코프 기반, 모두 공개 도메인):
| 서비스 | 도메인 | 신뢰 등급 |
|---|---|---|
| 검색 | search.naver.com | 공개 |
| 통합 로그인 (SSO) | nid.naver.com | **핵심 (인증축)** |
| 메일 | mail.naver.com | 민감 |
| 결제 | pay.naver.com | **핵심 (금전축)** |
| 클라우드 (MYBOX) | mybox.naver.com | 민감 |
| 커머스 | shopping.naver.com | 민감 |
여기서 핵심 관찰: **nid(인증)와 pay(결제)가 신뢰축**이다. 이 둘이 뚫리면 나머지 서비스로 권한이 전파된다.
---
## 2. OSINT 방법론 — 수동 정보 수집
> 아래 항목들은 모두 **수동(passive) 조사**다. 대상 서버에 직접 요청을 보내는 능동 스캔은 제외했으며, 네이버 BB 규정상 자동화 스캐닝은 금지된다.
### 2-1. 인증서 투명성 로그 (CT Log)
`crt.sh` 같은 공개 CT 로그에서 도메인의 발급 인증서 이력을 조회하면, 서브도메인 목록이 드러난다. 능동 스캔 없이 `dev.`, `staging.`, `internal-api.` 같은 비공개 의도 서브도메인을 찾을 수 있다.
```
조회 대상: %.naver.com
→ 발견 가능(예시): api-v2, staging-pay, legacy-auth, admin-internal ...
```
### 2-2. 공개 코드 저장소
GitHub/GitLab 공개 저장소에서 조직명·도메인을 검색하면 다음이 노출될 수 있다.
- 하드코딩된 API 엔드포인트
- 설정 파일의 내부 호스트명
- 과거 커밋에 남은 토큰(.git 히스토리)
### 2-3. 채용 공고 분석
채용 공고는 **기술 스택의 공개 명세서**다.
```
"Kafka, Kubernetes, Spring Boot 경험자"  → 백엔드 스택 추정
"AWS EKS, Terraform"                      → 인프라 구성 추정
"OAuth2, SAML 연동 경험"                  → 인증 아키텍처 추정
```
### 2-4. 모바일 앱 분석 (정적)
공개 배포된 APK/IPA를 정적 분석하면 API 엔드포인트, 도메인, SDK 구성이 드러난다. 앱을 실행해 트래픽을 발생시키는 것이 아니라, 패키지 내부 문자열을 읽는 수준이다.

## 3. 신뢰 경계와 STRIDE
### 3-1. 신뢰축 식별
```
        [외부 사용자]
              |
              v
        +----------+
        |  nid     | <- 인증축 (모든 서비스의 신뢰 기반)
        +----+-----+
             | 토큰 발급
   +---------+---------+---------+
   v         v         v         v
 [mail]   [pay]    [cloud]   [shop]
           ^ 금전축
```
nid가 발급한 토큰을 나머지 서비스가 신뢰한다. **nid가 뚫리면 전 서비스가 뚫린다.** 이것이 폭발 반경(blast radius)의 핵심이다.
### 3-2. STRIDE 매트릭스
STRIDE는 위협을 6가지로 분류하는 프레임워크다.
| 위협 | 의미 | 네이버 적용 가설 |
|---|---|---|
| **S**poofing | 신원 위장 | 토큰 위조로 타 사용자 가장 |
| **T**ampering | 변조 | 결제 요청 파라미터 조작 |
| **R**epudiation | 부인 | 감사 로그 부재로 행위 부인 |
| **I**nformation Disclosure | 정보 노출 | IDOR로 타 사용자 데이터 열람 |
| **D**enial of Service | 서비스 거부 | 인증 엔드포인트 자원 고갈 |
| **E**levation of Privilege | 권한 상승 | 일반 토큰 -> 관리자 권한 |

## 4. 취약점 가설 7종
> 모두 **구조상 가능한 가설**이며, 실제 존재 여부는 검증되지 않았다.
| # | 가설 | STRIDE | 우선순위 |
|---|---|---|---|
| V1 | SSO 토큰 검증 미흡 -> 계정 탈취(ATO) | S, E | **P0** |
| V2 | 결제 요청 IDOR -> 타 사용자 주문 열람 | I | P1 |
| V3 | 서브도메인 takeover (방치된 DNS) | S | P1 |
| V4 | API 버전 불일치 -> 구버전 인증 우회 | E | P2 |
| V5 | 과도한 CORS 정책 -> 토큰 탈취 | I | P2 |
| V6 | 비밀번호 재설정 토큰 예측 가능 | S | P2 |
| V7 | GraphQL introspection 노출 -> 스키마 유출 | I | P3 |

## 5. P0 심층 분석 — 계정 탈취(ATO) 체인
가장 우선순위가 높은 V1(SSO 토큰 계정 탈취)을 어택트리로 해부한다.
### 5-1. 토큰 생애주기
```
[로그인] -> [토큰 발급] -> [토큰 사용] -> [토큰 갱신] -> [토큰 폐기]
              ^              ^             ^            ^
            발급 결함      검증 결함      갱신 결함     폐기 결함
```
각 단계마다 결함 가설이 존재한다.
### 5-2. 어택트리 — 4가지 경로
```
                    [목표: 계정 탈취]
                          |
        +-------------+---+---+-------------+
        v             v       v             v
   [경로 A]       [경로 B]  [경로 C]     [경로 D]
   토큰 탈취      토큰 위조  세션 고정     재설정 악용
        |             |       |             |
   +----+----+   +----+---+  ...          ...
   v         v   v        v
 XSS로     CORS  약한 서명  알고리즘
 토큰탈취   악용  키 추측   confusion
                          (alg=none)
```
**경로 A — 토큰 탈취:** XSS나 과도한 CORS로 사용자 브라우저의 토큰을 빼낸다.
**경로 B — 토큰 위조:** JWT 서명 키가 약하거나, `alg=none` 같은 알고리즘 혼동(algorithm confusion) 공격이 가능하면 토큰을 직접 만든다.
**경로 C — 세션 고정:** 공격자가 미리 만든 세션 ID를 피해자가 쓰게 유도한다.
**경로 D — 재설정 악용:** 비밀번호 재설정 토큰이 예측 가능하거나 만료되지 않으면 탈취한다.
### 5-3. 킬체인 매핑 (MITRE ATT&CK)
| 단계 | ATT&CK 전술 | 가설 행위 |
|---|---|---|
| 정찰 | Reconnaissance (TA0043) | CT 로그·공개 저장소 수집 |
| 초기 접근 | Initial Access (TA0001) | 탈취/위조 토큰으로 로그인 |
| 권한 상승 | Privilege Escalation (TA0004) | 일반 -> 관리자 토큰 |
| 횡적 이동 | Lateral Movement (TA0008) | nid 토큰으로 pay/cloud 접근 |
| 수집 | Collection (TA0009) | 타 사용자 데이터 열람 |
### 5-4. 방어 컷셋 (Defense Cut Set)
어택트리의 각 경로를 차단하는 방어 지점이다.
| 공격 경로 | 방어 컷 |
|---|---|
| 경로 A (탈취) | HttpOnly 쿠키, 엄격한 CORS, CSP |
| 경로 B (위조) | 강한 서명 키, alg 화이트리스트, 키 회전 |
| 경로 C (고정) | 로그인 시 세션 재발급 |
| 경로 D (재설정) | 암호학적 난수 토큰, 짧은 만료 |
| **공통** | **교차서비스 이상 징후 상관분석** |
가장 강력한 단일 방어는 **인증축 강화 + 교차서비스 상관분석**이다. 하나의 토큰이 짧은 시간에 여러 서비스에서 비정상 패턴으로 쓰이면 탐지한다.

## 6. 위험 정량화 — CVSS와 DREAD
### V1 (P0 ATO)의 CVSS 평가 (가설)
```
CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:C/C:H/I:H/A:N
-> Base Score: 9.6 (Critical)
```
- **AV:N** (네트워크) — 원격 공격 가능
- **PR:N** (권한 불필요) — 인증 전 공격
- **S:C** (Scope Changed) — nid 결함이 다른 서비스로 전파
- **C:H / I:H** — 기밀성·무결성 심각
### DREAD 평가
| 항목 | 점수 (1-10) |
|---|---|
| Damage (피해) | 9 |
| Reproducibility (재현성) | 7 |
| Exploitability (악용 용이성) | 6 |
| Affected Users (영향 사용자) | 10 |
| Discoverability (발견 용이성) | 5 |
| **평균** | **7.4 (High)** |

## 7. 블루팀 — 탐지 엔지니어링
위협 모델의 진짜 가치는 **탐지 규칙으로 전환**될 때 나온다. 각 공격 가설마다 대응 탐지 신호를 정의한다.
| 공격 가설 | 탐지 신호 |
|---|---|
| 토큰 탈취 (V1-A) | 동일 토큰이 짧은 시간에 다른 IP/지역에서 사용 |
| 토큰 위조 (V1-B) | 서명 검증 실패 급증, `alg=none` 요청 |
| IDOR (V2) | 한 계정이 다수 타 사용자 ID 순차 접근 |
| 서브도메인 takeover (V3) | 방치 DNS 레코드 정기 스캔 |
| 재설정 악용 (V6) | 단시간 다수 재설정 요청 |
### Red-Blue 대칭 표
```
공격 행위              ->    탐지 신호
---------------------------------------------------
토큰 탈취 후 재사용     ->    지리적 불가능 이동 (impossible travel)
권한 상승 시도          ->    역할 변경 이벤트 모니터링
횡적 이동              ->    교차서비스 토큰 사용 상관분석
대량 데이터 수집        ->    비정상 조회량 임계치 알람
```

## 마치며
위협 모델링의 핵심은 세 가지다.
첫째, **폭발 반경으로 사고하라.** 멀티서비스 회사에서 중요한 건 결함의 개수가 아니라, 신뢰축 결함이 얼마나 멀리 전파되는가다. nid와 pay 같은 신뢰축을 우선 방어해야 한다.
둘째, **위협 모델은 탐지 백로그다.** 모든 공격 단계에는 대응하는 탐지 신호가 있다. 공격 가설을 세웠다면 그에 맞는 탐지 규칙을 짝지어야 한다.
셋째, **합법성이 모든 것에 우선한다.** 공개 정보, 구조적 가설, 버그바운티 규정 안에서만 이 작업은 정당하다. 활성 스캔이나 실제 침투는 정식 허가 없이는 절대 금지다. 실제 검증이 필요하다면 네이버 공식 버그바운티 채널을 통해, 규정 범위 안에서만 진행하고 공식 절차로 제보해야 한다.
